# Guide: Working with Climate Zone Tokens

## Prerequisites

- BIM token vault cloned locally (either `pointsav-design-system` or `woodfine-design-bim`).
- `app-orchestration-bim` deployed and configured to read from the vault.
- Git write access to the token vault repository.
- `ifctester` installed:

```sh
pip install ifctester
```

## Purpose

This guide covers how Climate Zone performance data is structured in the BIM token vault, how `app-orchestration-bim` resolves it at runtime, and how to add, validate, and verify a new climate zone overlay.

## Procedure

### 1. Locate the climate zone data file

Climate Zone data lives at:

```
<token-vault>/tokens/bim/climate-zones.dtcg.json
```

Each entry encodes one performance parameter for one element type in one climate zone:

```json
{
  "$type": "bim-climate-zone",
  "zone_id": "ASHRAE-5C",
  "ifc_class": "IfcWall",
  "parameter": "max_u_value",
  "required_value": 0.104,
  "unit": "Btu/h·ft²·°F",
  "source_standard": "ASHRAE 90.1-2022 Table 5.5-5",
  "$value": "climate-zone-spec"
}
```

### 2. Understand token resolution

When an element token is placed in a model, the token resolver:

1. Reads `climate-zones.dtcg.json` and finds all rows where `ifc_class` matches the placed element type and `zone_id` matches the project's assigned climate zone.
2. Binds the matched rows to the element as required property values — shown in the element's Climate Zone tab.
3. Projects resolved requirements into the IFC-SPF archive as `IfcConstraint` records linked to the element via `IfcRelAssociatesConstraint`.

The project site is linked to an `IfcSpatialZone` (PredefinedType: `USERDEFINED`, ObjectType: `CLIMATE_ZONE`) with the applicable zone identifier (e.g., `ASHRAE-5C`).

### 3. Add a new climate zone

To add performance data for a new climate zone:

1. Open `tokens/bim/climate-zones.dtcg.json` in the token vault.
2. Add rows for each element type and performance parameter in the new zone. Match the existing row structure exactly.
3. Commit to the token vault repository:

```sh
git add tokens/bim/climate-zones.dtcg.json
git commit -m "feat(climate-zones): add <zone-id> performance parameters"
```

4. If the zone requires an IDS constraint file, author the IDS file and add it to `climate/<zone-id>/` in the vault.
5. Restart `app-orchestration-bim` to pick up the new zone rows — the vault is read at startup.

### 4. Verify the update

Run `ifctester` with the IDS contract for the updated zone:

```sh
ifctester --ids climate-zones-<zone-id>.ids model.ifc --report json
```

The IDS contract file for a climate zone is generated by the token vault tooling (planned for v0.0.3). At v0.0.1, the IDS contract must be authored manually from the `climate-zones.dtcg.json` parameter rows.

## Expected outcome

`app-orchestration-bim` resolves climate zone requirements for all registered element tokens in the project's assigned zone. The element's Climate Zone tab displays all registered performance rows. The IFC-SPF archive contains `IfcConstraint` records for each resolved requirement.

## Verification

1. Open `app-orchestration-bim` and navigate to any element token whose `ifc_class` appears in `climate-zones.dtcg.json`.
2. Select the Climate Zone tab. Confirm rows appear for the registered zones.
3. For a newly added zone: confirm at least one row appears with the correct `zone_id`, `parameter`, and `required_value`.
4. Run `ifctester` against a model exported with the element:

```sh
ifctester --ids climate-zones-<zone-id>.ids exported.ifc --report json
```

A passing report confirms the constraint is correctly projected into the IFC-SPF archive.

## Rollback

**If new zone rows produce unexpected validation failures:** Open `climate-zones.dtcg.json`, remove the rows added in step 3, and commit the revert:

```sh
git revert HEAD
git push
```

Restart `app-orchestration-bim`. The new zone rows will no longer appear in token resolution.

**If `ifctester` reports a syntax error on the IDS file:** The IDS file is authoritative — the DTCG JSON is the source. Re-author the IDS file from the `climate-zones.dtcg.json` parameter rows and re-run verification. The DTCG JSON itself is unaffected.

**Note:** Adding climate zone rows is idempotent — re-running step 3 with the same data produces no change. There is no partial-state cleanup required if the procedure is interrupted before the commit.

## Registered climate zones at v0.0.1

| Zone ID | Description | Element types covered |
|---|---|---|
| ASHRAE-5C | Marine (Pacific Northwest, BC coast) | IfcWall, IfcSlab, IfcWindow |
| ASHRAE-6 | Cold | IfcWall, IfcSlab, IfcWindow |
| ASHRAE-7A | Very cold | IfcWall, IfcSlab |
| NBC-Zone-6 | Canadian cold climate | IfcWall, IfcSlab, IfcRoof |

Additional zones are planned for v0.0.3 as part of the BC RS-1 regulatory overlay set.
