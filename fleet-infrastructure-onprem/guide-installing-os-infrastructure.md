# Installing os-infrastructure

os-infrastructure is the sovereign operating system for PPN infrastructure nodes. It
installs on bare metal, cloud VMs, and development machines running QEMU.

**Before you start:** Purchase os-infrastructure at software.pointsav.com ($19 USDC).
Your download token unlocks all three artifact formats.

---

## Path 1 — Bare metal (USB boot)

Use this path to install on a physical machine: laptop, desktop, or decommissioned server.

**Requirements:**
- x86-64 machine with VT-x (Intel VT-x or AMD-V). Sandy Bridge i5 (2011+) or equivalent.
- 4 GB RAM minimum; 8 GB recommended.
- USB drive (8 GB or larger).
- Connectivity to the PPN genesis endpoint (ask your network administrator).

**Steps:**

1. Download `os-infrastructure-<ver>-x86_64.iso` from your software.pointsav.com download page.

2. Write the ISO to USB:
   ```
   # macOS / Linux — replace /dev/sdX with your USB device
   sudo dd if=os-infrastructure-<ver>-x86_64.iso of=/dev/sdX bs=4M status=progress
   # Or use Balena Etcher (GUI): select .iso → select USB → Flash
   ```

3. Boot the machine from USB. Select the USB device in your BIOS boot menu (usually F12
   or DEL at power-on).

4. The seL4 bootloader starts. You will be prompted for three values:
   - **Node name:** a short identifier for this machine (e.g., `laptop-a-1`)
   - **Genesis endpoint:** the WireGuard endpoint address of the genesis relay node
     (e.g., `34.x.x.x:51820`)
   - **Pairing code:** a one-time code generated by the os-network-admin operator

5. After pairing, the node registers in the PPN fleet. Verify from the fleet management
   node: the new node should appear within 30 seconds.

## Path 2 — Cloud VM import (GCP, DigitalOcean)

Use this path to add a cloud-hosted node to the PPN mesh.

**GCP import:**

```bash
# Upload the raw image to a GCS bucket
gsutil cp os-infrastructure-<ver>-x86_64.raw gs://your-bucket/

# Create a custom GCP image
gcloud compute images import os-infrastructure-<ver> \
  --source-file=gs://your-bucket/os-infrastructure-<ver>-x86_64.raw \
  --os=debian-11  # Talos/seL4 images use the closest matching OS family

# Create a VM from the image
gcloud compute instances create ppn-node-1 \
  --image=os-infrastructure-<ver> \
  --machine-type=n2-standard-2 \
  --min-cpu-platform="Intel Cascade Lake"  # VT-x required
```

Follow the same 3-question setup as the bare metal path via the VM's serial console.

**DigitalOcean / Hetzner:**
Upload the `.qcow2` artifact via the cloud provider's custom image import feature.
Boot the resulting droplet/server. Connect via VNC or serial console for initial setup.

## Path 3 — QEMU development (foundry-workspace or any Linux host)

Use this path to test os-infrastructure without a dedicated machine.

**Requirements:**
- Linux host with QEMU installed (`sudo apt install qemu-system-x86`)
- 4 GB RAM available for the VM
- No KVM required (TCG software emulation works but is slower)

**Steps:**

```bash
# Import the QCOW2 image
qemu-img info os-infrastructure-<ver>-x86_64.qcow2

# Boot under QEMU (TCG — no KVM)
qemu-system-x86_64 \
  -m 4096 \
  -cpu qemu64 \
  -drive file=os-infrastructure-<ver>-x86_64.qcow2,format=qcow2 \
  -nographic \
  -serial mon:stdio \
  -netdev user,id=net0,hostfwd=tcp::2222-:22 \
  -device virtio-net-pci,netdev=net0
```

Complete the 3-question setup via the QEMU serial console.

**Note:** TCG (software emulation) boots seL4 but is significantly slower than KVM.
Expect ~3–5 minutes for the Linux guest to fully initialize on TCG vs. ~15 seconds on KVM.

## Verifying the installation

From any node in the fleet, query the fleet management service:

```bash
curl http://<fleet-node-ip>:9203/nodes
```

The newly installed node should appear in the JSON response with its WireGuard IP and
resource availability within 30 seconds of the pairing code being entered.

## Troubleshooting

| Problem | Check |
|---|---|
| Pairing code not accepted | Verify code was generated within the last 5 minutes; codes expire |
| Node does not appear in fleet | Confirm WireGuard connectivity: `ping <genesis-wg-ip>` from new node |
| Boot fails on older hardware | Check VT-x is enabled in BIOS; disable Secure Boot |
| QEMU boots but network fails | Verify `wg0` is up: `wg show` inside Linux guest |
